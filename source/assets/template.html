1. Encounter rate λ (encounters per year)
Rate of bear encounters per year on this property. I_y is the estimated number of bear incursions per year per confident bear in the surrounding area; N_bears is the number of habituated/confident bears in the area; the food-availability modifier tunes the rate. Damage probabilities P(D_i|B,E) are no longer averaged into λ — they are applied per element in step 3.
λ = I_y · N_bears · food

2. Element-level encounter rate
The total encounter rate is split across the three element types according to the encounter share s_enc_i, which represents the fraction of bear visits targeting each element type.
λ_i = λ · s_enc_i

3. Annual loss without protection
Each encounter targeting element i causes damage with probability P(D_i|B,E), and a damaging event takes u_i units. Expected units damaged per year is therefore P(D_i|B,E) · λ_i · u_i, capped by the available stock N_i — you can't lose more than you have.
loss_i^noprot = min( P(D_i|B,E) · λ_i · u_i , N_i ) · c_i

4. Annual cost with protection (covering a share s_prot_i of N_i units)
Protection splits the stock into N_unprot = (1 − s_prot_i) · N_i unprotected units and N_prot = s_prot_i · N_i protected units. The bear's damaging "appetite" P(D_i|B,E) · λ_i · u_i is split between the two pools in proportion to their share of stock: fraction (1 − s_prot_i) of attempts target unprotected units and always succeed; fraction s_prot_i target protected units and succeed only when the measure has failed (P(¬M_j)). Each pool is then capped by its own available stock. At s_prot_i = 0 the unprotected term reduces to the no-protection loss in step 3.

Losses from unprotected units — appetite share (1 − s_prot_i) hits this pool, capped by N_unprot.
unprot_i = min( P(D_i|B,E) · λ_i · u_i · (1 − s_prot_i) , N_unprot ) · c_i

Losses from non-maintained infrastructure (measure fails) — appetite share s_prot_i reaches the protected pool but only succeeds with probability P(¬M_j); capped by N_prot.
residual_i = min( P(D_i|B,E) · λ_i · u_i · s_prot_i · P(¬M_j) , N_prot ) · c_i

Maintenance per year — recurring annual cost of keeping each protected unit functional. The per-unit maintenance rate is element-specific.
maint_i/yr = c_maintenance_i · N_i · s_prot_i

Installation (per cycle) — paid upfront when the measure is deployed, and re-paid every lifespan_j years when the measure reaches end-of-life and is replaced.
install_i = c_install_i · N_i · s_prot_i

5. Cumulative cost over T years
Without protection, cumulative cost grows linearly: T years of compensation. With protection, installation is paid in year 0 and re-paid every lifespan_j years (the chart shows step jumps at each re-investment). Each year between installations adds maintenance plus residual plus unprotected exposure. The crossover year — when the protected line drops below the unprotected one — is the payback period.
C_noprot(T) = T · Σ loss_i^noprot
C_prot(T) = (1 + ⌊T / lifespan_j⌋) · Σ install_i + T · Σ (unprot_i + residual_i + maint_i/yr)

Net benefit at T
Net(T) = C_noprot(T) − C_prot(T)

Model overview

This tool performs a cost-benefit analysis of two strategies for managing bear-caused damage on a single property over a fixed time horizon.

No protection — the owner pays compensation every time a damage event occurs. Costs grow linearly with time and bear pressure.

With protection — the owner installs a protective measure (electric fence, guardian dog, raised platform, etc.) covering a chosen share of each asset type. There is an upfront installation cost and a recurring annual maintenance cost, but losses from protected units are greatly reduced.

The model tracks three asset types — small livestock (sheep, goats, poultry), large livestock (cattle, horses), and agriculture (beehives, orchards) — each parameterised separately. The key output is the cumulative cost curve for both strategies, and the year at which protection becomes cheaper (the payback year).

Bear encounter pressure is driven by the number of habituated bears in the surrounding area and a baseline incursion rate. The Complete Model additionally exposes a food-availability modifier and per-element encounter distribution.

How to use the tool

All sliders write to a single shared model state. A change made on the Simple Model page is immediately reflected on the Complete Model page and in the Data Source table, and vice versa.

General context panel — set the time horizon, bear pressure, protective measure effectiveness and lifespan. These apply to all asset types.

Per-element panels — tune parameters for each asset type individually: how many units you have, what share is protected, compensation costs, and installation and maintenance costs of the measure.

Result cards — each asset type shows an annual cost breakdown and a cumulative cost chart. The overall chart aggregates all three types and displays the net benefit of protection at the chosen horizon.

Reset button — restores all sliders to the default values from the Excel workbook.

The verdict in the page header updates in real time and summarises whether protection or compensation is the better strategy at the chosen time horizon.

Data source

All default parameter values are read from bear_attack_model_variables.xlsx at build time and embedded into this file. The Data Source page displays the full variable dictionary and allows you to edit any adjustable value directly — changes take effect immediately in both model pages without rebuilding.

The workbook contains 31 variables organised into groups: bear pressure, environment modifiers, per-element damage probabilities, encounter shares, maintenance probability, unit counts, units damaged per event, compensation costs, protection coverage, installation and maintenance costs, measure lifespan, and analysis time horizon.

Two variables are derived and cannot be edited directly: s_enc_Ag (agriculture encounter share) is automatically computed as 1 − s_enc_SL − s_enc_LL so that encounter shares always sum to 100%; P(¬M_j) (measure failure probability) is 1 − P(M_j).

To change the default values permanently, edit the Excel workbook and regenerate the HTML by running python source/pipeline.py.

Simple vs Complete model

The tool offers two views of the same underlying model.

Simple Model — focuses on the parameters most likely to vary between properties: bear count, time horizon, protective measure quality, number of units, share protected, and costs. Advanced parameters (food availability modifier, encounter distribution, units damaged per event) are hidden so the interface stays uncluttered. Suitable for a quick first assessment or for users unfamiliar with the full model.

Complete Model — exposes every parameter in the model. Use this for sensitivity analysis, to understand how food availability or encounter distribution affects the result, or to calibrate the model carefully against observed data.

Both views share the same state — sliders changed in one view are visible in the other. The underlying formulas and calculations are identical; only the set of visible controls differs.
